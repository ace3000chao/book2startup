# OPSP Alignment

## R — 原文 (Reading)

> "If you want everyone on the same page, then you need this page first."
>
> — Verne Harnish, Scaling Up, 第8章 OPSP

---

## I — 方法论骨架 (Interpretation)

OPSP（One-Page Strategic Plan）是一页纸的战略执行工具，让组织内从CEO到一线员工都能在一张纸上看到：**我们要去哪里（BHAG/愿景）、我们怎么去（3-5年战略）、下一步是什么（年度主题和季度优先事项）、各自负责什么（各职能/部门的具体目标）**。

Scaling Up的核心观点：战略和执行之间最大的损耗不是方向错了，而是**信息在传递过程中失真**。CEO听到的战略和一线员工理解的战略往往相差甚远——不是因为不聪明，而是缺乏一个共同的一页纸参照物。

OPSP的核心价值是**对齐锚点**：每个人做什么决策之前，先看OPSP；每次跨部门冲突，先用OPSP对一遍；每次汇报，先说OPSP对应哪个优先事项。

不是一张图，是整个组织的GPS。

## A1 — 书中的应用 (Past Application)

### 案例 1: 某制造企业从战略混乱到全员一张纸
- **问题**: 公司有战略，但每个部门各做各的：销售追收入不顾利润、研发追功能不顾市场需求、生产追交付不顾质量。CEO觉得战略"已经传达了"，但执行完全走偏。
- **方法论的使用**: 用两天高管工作坊将10年BHAG、5年战略方向和本季度Top3优先事项浓缩到一页纸。然后将OPSP印成A3大小贴在公司每个会议室，每个汇报必须先说"我的汇报对应OPSP第X项"。
- **结论**: 不是战略没传达，是没有共同的参照物。
- **结果**: 三个月后部门间的资源争执减少60%，跨部门项目启动速度提升40%。

### 案例 2: 科技公司的远程团队对齐
- **问题**: 创始团队分散在3个城市，工程师和产品经理经常对"本季度最重要的事"理解不一致，导致资源争抢和返工。
- **方法论的使用**: 将OPSP设计成在线协作版本（Notion），每个职能的OKR直接嵌套在OPSP的季度优先事项下。每个OPSP优先项有且仅有1个总负责人。
- **结论**: 远程团队比面对面团队更需要OPSP——文字比口头更统一。
- **结果**: 产品路线图争议减少，季度末OKR完成率从55%提升到78%。

## A2 — 触发场景 (Future Trigger) ★

### 用户会在什么情境下需要这个 skill?

1. **战略落地效果不好时**：用户说"战略定了但执行走偏了"或"每个部门理解不一样"。
2. **全员大会或年会筹备时**：需要在全公司面前呈现清晰的战略方向和优先级。
3. **部门间资源争执时**：两个部门都说自己做的事情最重要，需要一个客观的对齐参照物。
4. **新员工入职时**：想让新员工快速理解公司方向和他在大图中的位置。
5. **汇报和沟通效率低下时**：每次汇报都要花大量时间解释背景，而非直接进入主题。

### 语言信号 (用户的话里出现这些就应激活)

- "战略落地……"
- "执行走偏……"
- "全员对齐……"
- "一页纸……"
- "部门各做各的……"
- "方向不一致……"

### 与相邻 skill 的区分

- 与 `Less Is More Priorities` 的区别：少即是多是优先级排序原则，OPSP是让全员看到和理解这些优先级的工具——原则需要载体才能传达。
- 与 `Fast Pulse Fast Growth` 的区别：快脉搏是节奏设计，OPSP是内容对齐——节奏对了但每个人看的内容不一样，组织同样无法对齐。

## E — 可执行步骤 (Execution)

当 skill 被激活后，agent 应按以下步骤执行:

1. **梳理OPSP的必备模块**
   - 标准OPSP包含（从上到下）：BHAG（10年愿景）→ 3年战略路径 → 年度主题 → 季度Top3优先事项 → 各职能/部门核心目标
   - 完成标准：确认用户当前各模块是否都有内容，特别是BHAG和季度优先事项

2. **制作或优化OPSP草稿**
   - BHAG检查：是否足够激动人心（10-25年不移的地标）？是否足够具体可衡量？
   - 季度优先事项检查：是否≤3项？每项是否只有1个总负责人？
   - 判停条件：若OPSP某模块内容清晰完整，跳过该模块的重建

3. **建立OPSP的使用仪式**
   - 每周例会开始前花5分钟对一遍OPSP：本周做的事情是否在Top3优先事项中？
   - 任何新增超过1人参与的项目必须先标注对应OPSP哪一项
   - 完成标准：产出一份包含OPSP落地使用规则的文件

## B — 边界 (Boundary) ★

### 不要在以下情况使用此 skill

- **战略本身尚未清晰时**：OPSP是战略的载体，如果公司战略还在激烈争论中，先对齐战略再制作OPSP。
- **小于10人的极早期团队**：信息量不大，面对面口头对齐足够，过早引入一页纸仪式感可能伤害敏捷性。
- **OPSP变成装饰品时**：如果OPSP做完了但从不被参考和使用，反而会产生"我们已经在对齐了"的虚假安全感。

### 作者在书中警告的失败模式

- OPSP做好了但没有嵌入日常使用流程——变成年度offsite的纪念品而非每天的决策参照物。

### 作者的盲点 / 时代局限

- OPSP在远程/异步工作环境中的使用方式（何时对、谁来对、不对的后果是什么）需要根据团队实际情况调整。

### 容易混淆的邻近方法论

- **`BHAG`**（Big Hairy Audacious Goal）：BHAG是OPSP最顶端的"目的地"描述，OPSP是承载BHAG及其他所有战略要素的整体框架——BHAG是内容，OPSP是形式。
